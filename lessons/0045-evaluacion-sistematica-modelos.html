7.2 — Evaluación sistemática de modelos (evals)

Lección 45 de 55 · ~16 min · Fase 7: Sistemas de IA en Producción

En el desarrollo de software tradicional, los tests automáticos te dicen si tu código funciona. En aplicaciones de IA, las evals (evaluaciones) hacen ese mismo trabajo — pero para el comportamiento del LLM. Sin evals, cada cambio al prompt es un salto de fe.

Qué son las evals

Una eval es un conjunto automatizado de casos de prueba que miden si el LLM produce el output deseado para un input dado. Al igual que los tests unitarios, se ejecutan antes y después de cada cambio. Al contrario que los tests unitarios, las respuestas del LLM rara vez son binarias — requieren criterios de evaluación más sofisticados.

Tipos de evals

TipoDescripciónMejor para
Exact matchLa respuesta debe ser exactamente XClasificación, extracción de datos estructurados
Contains / regexLa respuesta debe contener cierto patrónVerificar que se incluyen ciertos elementos
LLM-as-judgeUn LLM evalúa la calidad de la respuestaRespuestas abiertas, calidad de redacción, razonamiento
Human preferenceHumanos comparan dos respuestas y prefieren unaCalibrar la calidad percibida por usuarios
Task-specific metricsMétricas propias del dominio (precisión, recall, F1)Clasificación, extracción de entidades

LLM-as-judge: el evaluador más flexible

Cuando no puedes definir una respuesta correcta exacta, usa otro LLM para evaluar. El LLM juez recibe: el prompt original, la respuesta generada, y un rubric (criterios de calidad). Devuelve una puntuación y una justificación.

# Ejemplo de prompt para LLM-as-judge
system: "Eres un evaluador experto. Evalúa la respuesta según estos criterios:
  - Precisión factual (0-3)
  - Claridad y concisión (0-3)
  - Completitud (0-4)
  Total: 0-10. Devuelve JSON: {score: int, justification: str}"

user: "Prompt original: {original_prompt}
      Respuesta a evaluar: {model_response}"

Construir un dataset de evals

  1. Golden examples: 50-200 casos donde sabes la respuesta correcta (revisados por humanos).
  2. Edge cases: Inputs inusuales, ambiguos o que históricamente han causado problemas.
  3. Tests de regresión: Cada bug encontrado en producción se convierte en un test para que no vuelva a ocurrir.
  4. Distribución real: Los casos deben reflejar la distribución real de uso en producción, no solo casos ideales.

Eval-driven development

El flujo correcto es: define las evals primero, luego cambia el prompt. Así sabrás inmediatamente si el cambio mejoró o empeoró el comportamiento en todos los casos, no solo en el que estabas mirando.

Herramientas disponibles: Braintrust (potente, SaaS), PromptFoo (open source, CLI-first), LangSmith Evals (integrado con LangChain), Anthropic Console (para Claude).

Riesgo en producción Un sistema de IA en producción sin evals es un sistema que no sabes si funciona. Las regresiones silenciosas son el mayor riesgo en LLM applications.
