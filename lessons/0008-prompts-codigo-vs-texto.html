Prompts para código vs. prompts para texto

Fase 1 · Lección 8 de 55 · ~14 min · Prompt Engineering

Dos dominios, dos mentalidades

🔤 Prompts para texto

  • El tono y estilo importan tanto como el contenido
  • La audiencia debe estar explícita
  • La longitud y formato definen el output
  • La ambigüedad puede ser un recurso creativo
  • Temperatura 0.5–0.8 suele ser adecuada

💻 Prompts para código

  • El lenguaje y versión deben ser explícitos
  • El contexto del codebase importa mucho
  • Los tests y edge cases reducen errores
  • La ambigüedad produce bugs sutiles
  • Temperatura 0–0.2 para mayor consistencia

Plantilla de prompt para código

Estructura óptima para tareas de código [Lenguaje y versión]: Python 3.11, usando solo la stdlib
[Contexto]: Función dentro de un servicio FastAPI que maneja autenticación JWT
[Tarea]: Escribe una función `verify_token(token: str) → dict | None` que...
[Restricciones]: Sin dependencias externas, manejo de excepciones explícito
[Tests]: Incluye 3 casos de test con pytest (token válido, expirado, malformado)
[Formato]: Código Python con type hints, docstring conciso, comentarios solo donde no sea obvio
El principio de verificabilidad Para código, siempre pide tests o al menos especificaciones de comportamiento. El LLM puede generar código que parece correcto pero falla en edge cases. Los tests son tu red de seguridad.

Cómo dar contexto de codebase

El modelo no conoce tu proyecto. Incluye en el prompt: las interfaces relevantes, el framework usado, los patrones de nombrado del proyecto y el tipo de respuesta que se espera. Sin contexto, el modelo genera código genérico que puede no integrarse bien.
