Limitaciones reales y alucinaciones

Fase 0 · Lección 4 de 55 · ~12 min · Fundamentos Conceptuales

Por qué los LLMs alucinan

Una alucinación es cuando el modelo genera información plausible pero incorrecta con aparente confianza. No es un bug, es una consecuencia estructural de cómo funcionan: el modelo predice tokens probables, no verifica hechos.

Nunca confundas confianza con precisión Un LLM que escribe "La batalla de Hastings ocurrió en 1066" suena igual de confiado que uno que escribe una fecha inventada. La fluidez del lenguaje no es garantía de corrección factual.
¿Cómo puedo detectar si un LLM está alucinando en un output concreto?

Detectar las alucinaciones en un LLM (cuando el modelo inventa datos con total seguridad) en un output concreto es uno de los mayores retos de la IA actual. No existe un botón mágico, pero sí una combinación de estrategias técnicas y de ingeniería de prompts que puedes aplicar.

Aquí tienes los métodos más efectivos divididos por su enfoque:

1. Métodos basados en prompts (self-evaluation)

A veces, el propio LLM (o uno secundario) puede ayudarte a auditar la respuesta.

  • Prompt de auto-reflexión (self-reflection): Pasa el output generado a un segundo prompt y pídele al modelo que actúe como un corrector crítico. Ejemplo: "Analiza el siguiente texto y busca afirmaciones fácticas. Verifica si son lógicas o si entran en contradicción. Señala posibles inventos."
  • Técnica de los N-votos (self-consistency): Genera la misma respuesta 3 o 5 veces subiendo un poco la temperatura (por ejemplo, a 0.7). Si en las respuestas el modelo da datos radicalmente distintos (nombres, fechas, cifras), es muy probable que esté alucinando. La consistencia suele ser sinónimo de veracidad.

2. Métodos de verificación externa (ground truth)

La forma más segura es contrastar el output con una fuente de verdad indiscutible.

  • RAG (Retrieval-Augmented Generation): Obliga al modelo a extraer la información de una base de datos o documento que tú le proporciones. Para detectar la alucinación, extrae las afirmaciones del output y busca si existen literalmente en tu documento fuente.
  • Búsqueda en tiempo real (web search): Si el output contiene datos recientes o fácticos, utiliza una API de búsqueda para verificar las entidades (nombres, empresas, eventos) mencionadas.

3. Métodos técnicos y de probabilidad (métricas)

Si accedes al LLM a través de una API, puedes mirar detrás del capó.

  • Probabilidad de tokens: Al activar logprobs en la API, el modelo te devuelve la probabilidad que asignó a cada palabra. Si la probabilidad de datos críticos (como cifras) es baja o está muy dividida, el modelo puede estar adivinando.
  • Métricas de similitud semántica: Si tienes un texto de referencia (por ejemplo, un PDF origen), puedes usar embeddings para medir distancia semántica entre el output y la fuente. Herramientas como BERTScore o ROUGE ayudan a detectar desviaciones.

4. Herramientas y frameworks especializados

No tienes que programar esto desde cero. Ya existen librerías de código abierto diseñadas para evaluar y detectar alucinaciones en outputs.

Herramienta¿Qué hace?
RagasEspecializada en evaluar sistemas RAG. Mide la fidelidad del output respecto al contexto proporcionado.
DeepEvalFramework de pruebas unitarias para LLMs. Incluye tests específicos de alucinación, como HallucinationMetric.
TruLensEvalúa la triada RAG: relevancia del contexto, fundamentación (evitar alucinaciones) y relevancia de la respuesta.
Truco rápido de diseño Si estás diseñando el sistema, una forma simple de reducir y detectar alucinaciones es combinar el enfoque paso a paso con una regla estricta: "Si no sabes la respuesta basándote en los hechos, di 'No lo sé'". Si el modelo rompe esa regla y entrega un dato largo sin base clara, sospecha de inmediato.

Las limitaciones fundamentales

🗓️ Fecha de corte (Knowledge cutoff)

El modelo no tiene información posterior a su fecha de entrenamiento. Eventos, versiones de librerías o noticias recientes pueden ser desconocidos o incorrectos.

🔢 Aritmética y razonamiento simbólico

Los LLMs no calculan: predicen la respuesta probable. Para cálculos precisos necesitan herramientas externas (code interpreter, calculadora).

📏 Degradación con contextos muy largos

El modelo puede perder información relevante en las partes centrales de contextos muy largos. El fenómeno se conoce como "lost in the middle".

🔄 No aprende de la conversación

Cada llamada a la API es stateless. El modelo no recuerda conversaciones previas a menos que se incluyan explícitamente en el contexto.

⚠️ Sycophancy (servilismo)

Los modelos tienden a estar de acuerdo con el usuario incluso cuando el usuario está equivocado. Si afirmas algo con confianza, el modelo puede validarlo incorrectamente.

Estrategia práctica Para tareas críticas: pide siempre al modelo que cite fuentes, usa RAG para grounding factual (Fase 5.7), o verifica los outputs con herramientas externas. La IA es un colaborador, no una fuente de verdad.
