3.3 — Generación de tests unitarios e integración con IA

Fase 3 Lección 19 de 55 · ⏱ ~15 min · IA en el Entorno de Desarrollo

Por qué la IA destaca en la generación de tests

Los tests unitarios son uno de los casos de uso donde la IA aporta más valor de forma consistente. La razón: escribir tests es una tarea altamente estructurada y repetitiva que sigue patrones predecibles (Arrange–Act–Assert), algo en lo que los modelos de lenguaje son especialmente buenos.

Ventajas concretas de generar tests con IA:

  • Cubre rápidamente el happy path y los casos de error más comunes.
  • Genera variantes con diferentes valores de entrada sin esfuerzo manual.
  • Produce el boilerplate del test (imports, describe blocks, mocks básicos) en segundos.
  • Sugiere edge cases que el desarrollador podría no haber considerado.

Plantilla de prompt para tests de calidad

El contexto que le das al modelo determina la calidad de los tests generados. Un prompt efectivo debe incluir tres elementos obligatorios:

Genera tests unitarios con [Jest/Vitest/pytest/etc.] para la siguiente función.
Cubre: 1) el caso de éxito principal, 2) valores límite y edge cases,
3) casos de error y excepciones.

[Pegar la función completa aquí]

Edge cases específicos que quiero cubrir: [lista]

Los tres tipos de tests que la IA genera bien:

  1. Happy path: la función recibe inputs válidos y produce el output esperado.
  2. Edge cases: valores límite (0, -1, cadena vacía, array vacío, null, undefined, Infinity).
  3. Error cases: la función lanza la excepción correcta o devuelve el error apropiado ante inputs inválidos.

El riesgo crítico: tests que encodenan bugs

La IA genera tests observando el comportamiento actual del código. Si la función tiene un bug, el modelo puede escribir un test que espera el comportamiento incorrecto — pasando el test aunque el código esté mal.

Esto es especialmente peligroso porque:

  • Los tests parecen correctos y pasan en CI/CD.
  • Dan una falsa sensación de seguridad.
  • Si alguien arregla el bug más tarde, el test "falla" aunque el código sea ahora correcto.

La solución: revisa siempre las aserciones. Para cada expect(result).toBe(X), pregúntate: "¿Es X realmente el valor correcto según la especificación, o simplemente es lo que el código actual devuelve?"

💡 "Los tests generados por IA son un punto de partida, no el producto final. Añade tests para los casos de negocio que el modelo no conoce."
