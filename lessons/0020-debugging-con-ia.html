3.4 — Debugging y análisis de errores con IA

Fase 3 Lección 20 de 55 · ⏱ ~13 min · IA en el Entorno de Desarrollo

El principio del pato de goma aplicado a la IA

El rubber duck debugging es una técnica clásica: explicarle un bug a un objeto inanimado (o a un compañero) con el máximo detalle posible. El acto de articular el problema en voz alta frecuentemente revela la solución antes de recibir respuesta.

La IA es el pato de goma definitivo: disponible 24/7, nunca se aburre de tus preguntas, y además sí puede responder. Más importante, al forzarte a describir el problema con precisión —qué pasa, qué esperabas que pasara, qué ya probaste— activas el mismo mecanismo cognitivo que hace funcionar la técnica original.

Esta es la razón por la que la estructura del prompt de debugging importa tanto: no es solo para el modelo, es también para ti.

Qué incluir en un prompt de debugging

Un prompt de debugging incompleto genera respuestas genéricas. Los cuatro ingredientes esenciales:

  1. El mensaje de error: copia el stack trace completo, no solo la primera línea.
  2. El código relevante: la función o bloque donde ocurre el error, con contexto suficiente.
  3. Lo que ya intentaste: evita que el modelo sugiera lo que ya sabes que no funciona.
  4. El entorno: versión del lenguaje/framework, sistema operativo si es relevante, datos de entrada que reproducen el error.

Plantilla lista para usar:

Tengo este error: [stack trace completo]

El código relevante es:
[código]

Ya intenté: [lista de lo probado]

¿Cuál es la causa más probable y cómo lo solucionaría?

Superpoderes de la IA en debugging

La IA destaca especialmente en categorías de bugs donde el patrón del error es reconocible aunque el desarrollador no tenga experiencia con esa tecnología específica:

  • Expresiones regulares: interpretar regex complejas, detectar por qué no captura el patrón esperado, reescribir con explicación.
  • SQL y optimización de queries: analizar planes de ejecución, detectar N+1, sugerir índices, reescribir subqueries como JOINs.
  • CSS y layout: problemas de flexbox, grid, z-index, especificidad, box model.
  • Ficheros de configuración: YAML con indentación incorrecta, JSON malformado, variables de entorno mal referenciadas, conflictos en webpack/vite/tsconfig.
  • Errores de tipos: mensajes de TypeScript crípticos, discrepancias de tipos genéricos, aserciones incorrectas.
💡 "Incluir siempre 'ya intenté X' evita que el modelo sugiera lo que ya sabes que no funciona."
