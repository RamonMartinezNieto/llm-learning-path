7.1 — Fine-tuning vs. prompting: cuándo usar cada enfoque

Lección 44 de 55 · ~15 min · Fase 7: Sistemas de IA en Producción

Una de las preguntas más frecuentes en equipos que llevan aplicaciones de IA a producción: "¿Cuándo debería hacer fine-tuning en lugar de seguir mejorando los prompts?" La respuesta honesta es que la mayoría de las veces, no necesitas fine-tuning. Pero cuando sí lo necesitas, marca una diferencia real.

El framework de decisión

Sigue este orden estrictamente:

  1. Prompting primero: Itera el system prompt, añade few-shot examples, prueba diferentes framing. El 70% de los problemas se resuelven aquí.
  2. RAG si el problema es conocimiento: Si el modelo no sabe algo porque no está en su entrenamiento, RAG es la solución, no fine-tuning.
  3. Fine-tuning solo si: Tienes >1000 ejemplos de alta calidad, necesitas estilo/formato ultra-consistente, y la latencia/coste a escala lo justifica.

Qué puede (y no puede) hacer el fine-tuning

Fine-tuning SÍ puedeFine-tuning NO puede
Consistencia de formato/estiloAñadir conocimiento factual nuevo
Seguir instrucciones específicas más fielmenteActualizar información desactualizada
Reducir prompts largos (el comportamiento queda "horneado")Eliminar alucinaciones sobre hechos
Ajustar tono y vocabulario de forma consistenteRazonar mejor sobre dominios nuevos sin datos

La regla de oro: haz fine-tuning sobre comportamientos (cómo responde), no sobre conocimiento (qué sabe). Para conocimiento, usa RAG o context injection.

Comparativa de costes

AspectoPromptingFine-tuning
Coste inicialBajo (solo iteración)Alto (preparación de datos + entrenamiento)
Coste por llamadaNormal + tokens del promptPuede ser menor (menos tokens de sistema)
Tiempo hasta producciónDíasSemanas (preparar datos, entrenar, evaluar)
MantenimientoEditar el promptRe-entrenar con nuevos datos si el comportamiento cambia
Ejemplos mínimos útiles5-20 (few-shot)500-1000+ (entrenamiento)

Cuándo el fine-tuning sí tiene sentido

  • Necesitas que el modelo responda siempre en un formato JSON muy específico con campos anidados complejos
  • Tienes una jerga de dominio muy específica (legal, médico, código propietario) y >1000 ejemplos etiquetados
  • El prompt de sistema es tan largo que impacta significativamente el coste a escala
  • Latencia extremadamente crítica y cada token cuenta
Regla del 90% El 90% de los casos de uso no necesitan fine-tuning. Antes de plantearlo, asegúrate de haber agotado las técnicas de prompting y RAG.
