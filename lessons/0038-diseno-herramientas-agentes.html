6.2 — Diseño de herramientas (tools) para agentes

Lección 38 de 55 · ~16 min · Fase 6: Código Agéntico

Si el bucle ReAct es el motor del agente, las herramientas son sus manos. El agente solo puede actuar sobre el mundo mediante las herramientas que le proporcionas, y la calidad de esas herramientas determina en gran medida la calidad del agente. Diseñar buenas herramientas es la habilidad de ingeniería más importante en el desarrollo de agentes.

Qué hace buena a una herramienta

Una buena herramienta tiene cuatro propiedades fundamentales:

  • Enfocada: Hace una sola cosa bien. No intentes crear una herramienta do_everything.
  • Bien descrita: Su nombre, parámetros y descripción explican al LLM exactamente cuándo y cómo usarla. La descripción es el "contrato" entre el modelo y la herramienta.
  • Fiable: Devuelve resultados consistentes y predecibles. Los errores siempre vienen en formato estructurado.
  • Retorna datos estructurados: JSON, no texto libre. El modelo necesita parsear el resultado para razonar sobre él.

Principios de diseño

Single Responsibility: Cada herramienta tiene un propósito único. En lugar de una herramienta file_operations con un parámetro mode, crea read_file y write_file por separado.

Errores predecibles: Cuando una herramienta falla, el error debe ser informativo y estructurado. El modelo necesita saber si falló por permisos, por archivo no encontrado, o por un error de red — no solo "Error 500".

Operaciones atómicas: Cada herramienta completa una operación indivisible. Evita herramientas que realicen múltiples efectos secundarios en una sola llamada.

Ejemplo: herramientas para un coding agent

# Buenas herramientas para un coding agent

read_file(path: str) -> {"content": str, "lines": int}
write_file(path: str, content: str) -> {"success": bool, "bytes_written": int}
run_command(cmd: str, timeout: int) -> {"stdout": str, "stderr": str, "exit_code": int}
search_code(query: str, path: str) -> {"matches": [{"file": str, "line": int, "text": str}]}
create_pr(title: str, body: str, branch: str) -> {"url": str, "number": int}

Observa que cada herramienta devuelve un objeto JSON con campos bien nombrados. El modelo sabe exactamente qué esperar y puede razonar sobre los resultados.

Las descripciones son la interfaz principal

El LLM no ejecuta las herramientas directamente — decide cuál usar basándose en sus descripciones. Una descripción pobre hace que el modelo use la herramienta en el momento equivocado o con los parámetros incorrectos. Ejemplo de buena descripción:

{
  "name": "search_code",
  "description": "Busca texto o patrones en archivos de código del repositorio.
    Usar cuando necesites encontrar dónde se define una función, dónde se usa
    una variable, o qué archivos contienen cierta lógica. NO usar para leer
    el contenido completo de un archivo específico (usa read_file para eso).",
  "parameters": {
    "query": "El texto o patrón a buscar",
    "path": "Directorio donde buscar (opcional, por defecto: raíz del repo)"
  }
}

Advertencia Una herramienta mal descrita o con efectos secundarios inesperados puede hacer que el agente falle de formas difíciles de depurar. El diseño de herramientas es la parte más crítica de la ingeniería de agentes.
