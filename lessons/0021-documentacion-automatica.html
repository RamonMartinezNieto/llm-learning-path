3.5 — Documentación automática de código con IA

Fase 3 Lección 21 de 55 · ⏱ ~11 min · IA en el Entorno de Desarrollo

Tipos de documentación que la IA genera bien

La documentación es una de las tareas más postergadas en el desarrollo de software. La IA elimina gran parte de la fricción inicial porque puede producir documentación estructurada a partir del código en segundos.

Tipos donde el modelo aporta más valor:

  • Docstrings: Python (Google style, NumPy, reStructuredText), con parámetros, tipo de retorno y descripción de excepciones.
  • JSDoc / TSDoc: comentarios tipados para JavaScript y TypeScript, con @param, @returns, @throws, @example.
  • Secciones de README: instalación, uso, ejemplos, configuración, contribución.
  • Comentarios inline: explicación de lógica compleja, regex difíciles, algoritmos no triviales.
  • Changelogs: generados a partir del diff de commits.

Plantilla de prompt para docstrings y JSDoc

Para obtener documentación en el formato correcto, el prompt debe especificar explícitamente el estilo. Un formato genérico produce documentación que no se integra con las herramientas del proyecto (TypeDoc, Sphinx, pdoc, etc.).

Genera la documentación JSDoc completa para esta función TypeScript.
Usa el formato TSDoc. Incluye: @param con tipos, @returns,
@throws si aplica, y un @example con un caso de uso real.

[Pegar la función aquí]
Cuándo NO sobre-documentar:
  • Evita comentarios que repiten lo que el nombre ya dice: // incrementa el contador en counter++ es ruido, no señal.
  • El código legible no necesita comentarios de traducción. Necesita comentarios de intención.
  • La IA tiende a documentar el qué (más fácil de inferir del código); tu tarea es asegurarte de que también documente el por qué cuando no sea obvio.

Documentación viva: que evolucione con el código

El mayor riesgo de la documentación no es no tenerla, sino tener documentación desactualizada. Una estrategia efectiva es adoptar el enfoque de "documentación viva":

  • Al modificar una función, incluye en tu prompt al agente: "Actualiza también el JSDoc para que refleje este cambio."
  • En revisiones de PR, pide: "¿La documentación existente sigue siendo precisa tras estos cambios?"
  • Para módulos sin documentación, usa: "Genera docstrings en formato [X] para todas las funciones públicas de este fichero."

La clave es tratar la documentación como parte del ciclo de desarrollo, no como una tarea separada al final del sprint.

💡 "El mejor comentario explica el 'por qué', no el 'qué'. La IA tiende a documentar el 'qué' — revisa que incluya el 'por qué' cuando sea importante."
