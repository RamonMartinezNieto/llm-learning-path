6.4 — Agentes de código: desde issue a PR de forma autónoma

Lección 40 de 55 · ~16 min · Fase 6: Código Agéntico

Los coding agents representan uno de los avances más prácticos de la IA agéntica. A diferencia de un asistente de código que sugiere fragmentos, un coding agent puede leer un repositorio real, entender su contexto, planificar cambios, escribir el código, ejecutar los tests y crear una Pull Request — todo de forma autónoma o con supervisión mínima.

Arquitectura de un coding agent

El ciclo de un coding agent típico tiene estas etapas:

  1. Comprensión del codebase: Lee archivos relevantes, entiende la estructura del proyecto, localiza el código relacionado con la tarea.
  2. Planificación: Define qué archivos modificar, en qué orden, y qué tests escribir o actualizar.
  3. Implementación: Escribe o modifica código archivo por archivo, manteniendo coherencia entre ellos.
  4. Verificación: Ejecuta los tests, corrige errores, hace linting. Itera hasta que pasan.
  5. Entrega: Crea la PR con descripción de los cambios, qué se testó y por qué se tomaron ciertas decisiones.

Ejemplos en producción: Claude Code y GitHub Copilot Agent

Claude Code (Anthropic) es un agente de terminal que opera sobre el directorio local. Lee el codebase completo, razona sobre los cambios necesarios y los implementa. Especialmente efectivo para: corregir bugs con stack traces, implementar funciones con specs claras, refactorizar código con tests existentes.

GitHub Copilot Agent Mode integra el agente directamente en VS Code, con acceso al workspace del editor. Puede leer errores de la consola y el estado del linter en tiempo real para corregir iterativamente.

El desafío de la coherencia entre archivos

El mayor reto de los coding agents no es escribir código correcto en un archivo — es mantener coherencia cuando hay que modificar múltiples archivos interdependientes. Un cambio en una interfaz de TypeScript debe propagarse a todos los archivos que la implementan. Si el agente no tiene un plan global del alcance del cambio, puede "arreglar" un archivo y "romper" otro.

La solución es la planificación explícita: antes de escribir código, el agente debe generar un plan con todos los archivos afectados y el orden de modificación.

Human-in-the-loop: cuándo pedir confirmación

Los buenos coding agents incluyen checkpoints donde piden confirmación antes de acciones de alto impacto:

  • Antes de modificar más de N archivos
  • Antes de ejecutar comandos destructivos (rm, migraciones de base de datos)
  • Cuando los tests fallan repetidamente y el agente no encuentra la causa
  • Cuando el scope de la tarea es mayor de lo esperado
Recomendación práctica Los coding agents actuales son más útiles para tareas bien definidas (fix this bug, implement this function) que para arquitectura abierta. Define el scope claramente.
