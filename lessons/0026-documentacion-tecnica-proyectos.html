4.4 — Automatización de documentación técnica de proyectos

Fase 4 · Lección 26 de 55 · ~12 min · IA en el Ciclo DevOps

Qué documentación técnica genera bien la IA

La documentación técnica es uno de los artefactos más abandonados del desarrollo de software. No por falta de intención, sino porque documentar consume tiempo que siempre parece más urgente dedicar a escribir código. La IA cambia esta ecuación drásticamente: el costo de generar documentación cae de horas a minutos.

Tipos de documentación donde la IA produce resultados de alta calidad:

  • README: descripción del proyecto, instrucciones de instalación, ejemplos de uso, badges de estado.
  • Documentación de arquitectura: diagramas textuales (Mermaid, PlantUML), descripciones de componentes y sus responsabilidades.
  • Architecture Decision Records (ADRs): dado el contexto de una decisión técnica, la IA puede generar el ADR con el formato estándar.
  • Changelogs: a partir del historial de commits o PRs, generar un CHANGELOG.md estructurado siguiendo Keep a Changelog.
  • Documentación de API: a partir de una especificación OpenAPI, generar guías de uso con ejemplos y casos de error.
  • Docstrings y comentarios inline: documentación a nivel de función y módulo.

Prompt template para documentación README

El README es el documento más importante de cualquier proyecto. Aquí un prompt efectivo para generarlo:

Genera un README.md completo para este proyecto. Incluye:
- Badge de licencia, tests y versión
- Descripción clara del problema que resuelve (no solo "qué es")
- Requisitos y dependencias
- Instalación paso a paso
- Ejemplo de uso mínimo (quickstart)
- Configuración disponible (tabla con todas las opciones)
- Cómo contribuir
- Licencia

Contexto del proyecto:
- Nombre: [nombre]
- Stack: [tecnologías]
- Propósito: [qué problema resuelve, para quién]
- Principales features: [lista]

Docs as Code: mantener la documentación sincronizada

El mayor problema de la documentación no es crearla, sino mantenerla actualizada. La solución moderna es el enfoque docs as code: la documentación vive en el mismo repositorio que el código, se versiona con git y se actualiza en el mismo PR que cambia el código.

La IA encaja perfectamente en este flujo:

  • En cada PR: pide a la IA que identifique qué documentación existente queda desactualizada por los cambios del PR.
  • Generación de changelog: automatiza la generación del changelog a partir de los mensajes de commit usando la IA como intermediario entre commits crudos y texto legible.
  • Actualización de ADRs: cuando se toma una decisión técnica, generar el ADR inmediatamente con el contexto fresco.
💡 Momento óptimo para documentar

El mejor momento para documentar es justo después de escribir el código, mientras el contexto está fresco. La IA puede generar el 80% — tú aportas el contexto y las decisiones de diseño.

La división de responsabilidades óptima entre humano e IA en la documentación:

  • IA genera: estructura, boilerplate, ejemplos de código, tablas de configuración, descripciones de funciones.
  • Humano aporta: decisiones de diseño, trade-offs considerados, razones por las que se eligió un enfoque sobre otro, contexto histórico del proyecto.

👨‍🏫
Nota del instructor El framework Diátaxis es uno de los marcos más útiles para pensar en documentación técnica. Antes de pedir a la IA que genere documentación, decide qué tipo necesitas: ¿un tutorial (aprendizaje guiado), una how-to guide (tarea específica), una reference (información técnica precisa) o una explanation (comprensión conceptual)? Cada tipo tiene una estructura diferente y la IA necesita ese contexto para generar algo útil.
