6.3 — Bucles de agente: planificación, ejecución, corrección

Lección 39 de 55 · ~16 min · Fase 6: Código Agéntico

Entender el bucle de un agente no es solo entender ReAct — es entender cómo implementar ese bucle de forma robusta en producción. Un bucle bien diseñado planifica, ejecuta, corrige y sabe cuándo detenerse. Un bucle mal diseñado gasta dinero, entra en ciclos infinitos o falla silenciosamente.

La estructura del bucle

En pseudocódigo, el bucle de un agente típico es:

def agent_loop(task, max_iterations=20, budget_tokens=50000):
    context = [system_prompt, task]
    iterations = 0
    total_tokens = 0

    while not is_complete(context) and iterations < max_iterations:
        response = llm.call(context)
        total_tokens += response.tokens_used

        if total_tokens > budget_tokens:
            return error("Presupuesto de tokens agotado")

        if response.type == "final_answer":
            return response.content

        if response.type == "tool_call":
            result = execute_tool(response.tool, response.args)
            context.append(response)   # pensamiento del agente
            context.append(result)     # observación
            checkpoint(context)        # guardar progreso

        iterations += 1

    return error("Máximo de iteraciones alcanzado")

Stopping conditions: cuándo parar

Las condiciones de parada son críticas y deben definirse antes de desplegar cualquier agente:

  • Tarea completada: El agente declara explícitamente que ha terminado con una respuesta final.
  • Máximo de iteraciones: Límite duro de pasos (ej. 20 iteraciones). Previene bucles infinitos.
  • Presupuesto de tokens: Límite de tokens totales consumidos en el ciclo.
  • Umbral de errores: Si N herramientas seguidas fallan, escalar al usuario en vez de reintentar indefinidamente.
  • Timeout absoluto: Tiempo máximo de ejecución en segundos/minutos.

Manejo de fallos: el agente debe decidir

Cuando una herramienta falla, el agente no debe simplemente parar. Debe razonar sobre el fallo:

Thought: La herramienta search_web devolvió un error de timeout.
         Tengo tres opciones:
         1. Reintentar con una query más simple
         2. Usar mi conocimiento de entrenamiento si el dato no es crítico
         3. Informar al usuario que no puedo completar esta parte

Action: search_web("Kafka release date site:apache.org")  # query más específica

El contexto del fallo se incluye en el siguiente paso de razonamiento. El agente no ignora el error — lo integra en su plan.

Checkpointing: guardar progreso intermedio

Para tareas largas, guardar el estado del contexto tras cada iteración permite:

  • Reanudar si el proceso se interrumpe (crash, timeout de red)
  • Auditar qué hizo el agente en cada paso
  • Detectar bucles donde el agente repite la misma acción

Estrategias de fallback

SituaciónEstrategia recomendada
Herramienta falla 1 vezReintentar con parámetros ajustados
Herramienta falla 3 vecesIntentar herramienta alternativa
Sin herramienta alternativaResponder con lo que se tiene + notificar limitación
Agente en bucle (repite acción)Detectar y forzar stop; escalar al usuario
Presupuesto agotadoDevolver respuesta parcial con advertencia
Advertencia crítica Sin stopping conditions bien definidas, un agente puede entrar en bucles infinitos o acumular costes enormes. Siempre define el presupuesto máximo de iteraciones y tokens.
